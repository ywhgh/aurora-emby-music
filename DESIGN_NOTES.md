# Emby Music Web Design Notes

目标：做一个轻量、响应式、体验完整的 Emby 音乐服务器网页客户端。

## 产品方向

- 浅色、干净、红色点缀的音乐客户端气质。
- 桌面端固定播放控制区、播放队列、播放页封面/歌词分区、动态播放主题感和克制的工具栏排版。
- 自托管音乐库的专辑/歌曲浏览、PWA 体验和清晰信息层级。
- 服务器音乐库管理、搜索、排序、长列表浏览和错误状态。
- 移动端友好的播放控制、队列入口和专辑/艺人浏览。

## 用户角度

- 首屏必须先让用户完成 Emby 登录，不把未连接状态伪装成完整应用。
- 登录入口要容忍用户从 Emby Web 复制地址，服务器根地址、`/emby` 和 `/web/index.html` 都应归一化到可用接口地址。
- 服务器地址输入不使用浏览器原生 URL 校验拦截提交；错误应该由应用给出明确提示，而不是让用户停在表单里猜原因。
- 登录页要提供独立的服务器测试动作，先验证地址、网络和 Emby 版本，再进入账号认证。
- 登录页事件绑定必须早于内部播放器/资料库初始化；内部初始化失败时也要给出可见错误，不能表现为按钮无响应。
- 登录页需要直接暴露运行版本、复制诊断和清缓存重载入口，排查旧 service worker、旧脚本和地址规范化问题不能依赖用户先进入应用内部。
- 登录表单必须有内联兜底保护；主脚本未加载时也不能把 `password` 等字段通过 GET 拼进地址栏，并且仍要支持测试服务器和基础认证登录。
- 如果用户已经落到了带 `serverUrl`、`username`、`password` 的地址栏状态，页面必须立即清理敏感参数，并接管为一次正常登录尝试，避免用户以为已经提交却停在登录页。
- 主脚本不能完全依赖内联兜底脚本传递登录参数；如果地址栏里仍有登录参数，主脚本也要二次接管并清理。
- 连接动作同时覆盖表单提交和按钮点击，避免浏览器、自动填充或旧页面状态导致看起来点击无响应。
- 自托管部署要支持预填或锁定服务器地址；给家人或固定设备使用时，用户只需要输入账号密码。
- 连接后应马上看到音乐库规模、最近歌曲和可播放入口。
- 多音乐库服务器要允许用户切换当前音乐库分区；选择分区后，搜索、分页和收藏读取都应围绕同一个范围，并记住上次选择，避免列表内容混杂。
- 音乐库筛选应覆盖整理资料库的常见状态，包括收藏与未收藏；用户不应为了找未整理内容只能靠搜索。
- 排序不应只有固定方向；长列表里用户需要在默认、升序和降序之间切换，且默认方向应保留音乐软件常识：最近/时长倒序，标题/艺人/专辑正序。排序字段和方向都要保存，并在设置/诊断里可见。
- 长列表需要密度选择；桌面端整理大量歌曲时，用户应能切到紧凑模式减少滚动，默认仍保持舒适阅读。
- 首页应承担“每日入口”的角色：继续收听、歌单、收藏和最近添加歌曲都要有清晰路径。
- 已恢复的本地队列应在首页可见，用户重新打开 PWA 时能直接继续上次曲目或进入队列确认。
- 恢复队列必须显示保存时间，避免用户误以为很久以前的本地队列是当前服务器状态。
- 歌曲列表不能只是播放入口；艺人和专辑文本应可跳转，帮助用户顺着音乐关系继续浏览。
- 正在播放页也应提供同样的艺人/专辑跳转，不让沉浸播放成为浏览断点。
- 详情页返回必须保留来处上下文，避免从首页、收藏页或嵌套详情进入后被强行带到顶层列表。
- 长列表页面切换必须保留滚动位置；用户从音乐库中段进入详情再返回，不应被迫重新滚动定位。
- 所有筛选状态都要可见、可清除，避免用户不知道列表为什么变少；常用整理筛选要能按服务器用户保存，刷新后不丢上下文。
- 筛选状态要支持单项移除；当搜索、风格、年份、音质叠加时，用户不应只能清空全部重新选。
- 全局筛选应尽量作用于所有歌曲列表，包括最近播放；页面之间的筛选语义不能互相打架。
- 大库搜索不能只依赖首屏已加载数据；输入稳定后应请求 Emby 搜索并合并结果。
- 大库浏览还需要轻量筛选；风格/流派和年份是音乐库里常见的第二入口。
- 自托管音乐库用户会关心文件格式和音质；格式/码率信息应轻量可见，并能作为筛选入口，但不能压过歌曲标题和封面。
- 歌单是日常听歌入口，必须作为一等页面，而不是藏在全部歌曲筛选里。
- 创建歌单应从歌单页直接发起，先支持空音频歌单，再通过歌曲行“添加到歌单”填充内容。
- 歌曲行要能直接加入已有歌单，减少“先记住歌曲，再进入歌单管理”的来回跳转。
- 歌单详情需要能直接调整歌曲顺序；上移、下移适合精细整理，拖拽适合快速重排。
- 歌单详情里的移除动作必须只删除歌单条目，不删除媒体库文件；按钮文案和实现都要避免制造危险误解。
- 收藏不能只停留在单曲层级；专辑、艺人和歌单都需要可见的收藏入口。
- 收藏页要像音乐客户端的“我的收藏”，聚合歌单、专辑、艺人和歌曲，而不是只列歌曲。
- 应用能力增多后需要设置页承接状态和本地操作，避免把清缓存、诊断、快捷键散落在各处。
- 设置页需要能主动测试当前连接，用户登录后遇到加载/播放问题时，不应该只能回到登录页验证服务器；当前音乐库分区、排序状态和筛选状态也应在设置页可见，方便判断列表范围和顺序。
- 播放失败要区分可忽略的浏览器中断和真实错误，并尝试转码降级。
- 播放源策略要可控；用户应能在自动、仅直连和优先转码之间切换，方便处理不同格式、浏览器和服务器转码能力差异。
- 当前播放上下文要暴露真实播放状态；用户应能直接看到直连还是转码、当前码率和策略，而不是只能在失败后猜测。
- PWA/浏览器窗口标题也属于播放上下文；任务栏、Alt-Tab 和独立窗口里应能看到当前歌曲和播放状态。
- 关键错误要有恢复入口：重试、切到队列、查看设置或刷新，而不是只展示一行不可操作文字。
- 浏览器离线和网络恢复都要可见；PWA 独立窗口里用户不一定能看到浏览器网络提示，应用内需要给出测试连接和刷新入口。
- 大音乐库不能一次性全量加载；先展示首批内容，再允许渐进加载更多。
- 大音乐库浏览要保留上下文：页面滚动位置只保存在当前运行状态内，避免本地持久化太多 UI 噪音。
- 列表接近底部时自动继续加载，手动按钮作为可见状态和兜底。
- 服务端搜索使用防抖和请求序号，旧搜索结果不会覆盖新输入。
- 自托管网页应支持安装到桌面/主屏幕，但离线能力只承诺应用外壳，不伪装服务器内容可离线。
- 收藏应和 Emby 同步；失败时回滚 UI，避免用户以为已经保存。
- 最近播放先保证本地可靠，再考虑跨设备同步。
- 播放器应覆盖常用收听控制：顺序、循环、单曲、随机、音量和静音。
- 睡眠定时属于音乐播放器的日常控制，应在底部播放器和正在播放页都可触达，到点只暂停播放，不清空队列。
- 设置页承接播放源策略和转码码率，避免把排障能力塞到底部播放器造成日常听歌噪音。
- 队列要能被用户直接管理：添加、上移、下移、拖拽排序、移除，而不是只能被动显示。
- 长队列里要能一键定位当前播放歌曲，当前状态不应只靠用户手动滚动寻找。
- 长队列整理要支持清除已播内容；用户想缩短临时队列时，不应该被迫清空当前播放或重新排队。
- 队列不仅要显示总量，还要显示当前曲目之后的剩余歌曲和剩余时长，帮助用户判断还会播多久。
- 队列随机重排应只作用于当前曲目之后的待播歌曲，不打断当前播放，也不偷偷切换播放模式。
- 队列整理也只作用于当前曲目之后的待播歌曲，按艺人、专辑、碟号、曲号和歌名排序，帮助用户从随机/拖拽状态回到可预期顺序。
- 歌曲行需要提供“下一首播放”，这是整理临时收听队列的高频动作，不能强迫用户先进队列页再拖动。
- 队列不应因为刷新或 PWA 重开丢失；本地恢复队列能降低误操作成本。
- 恢复后的播放进度要在底部播放器和正在播放页可见，用户需要知道点击播放会从哪里继续。
- 专辑、艺人和歌单详情需要支持整组追加队列，避免用户为了不中断当前播放只能逐首添加。
- 专辑、艺人和歌单详情还需要整组下一首播放，保持原顺序插入当前曲目之后，比逐首点 `+1` 更符合临时收听场景。
- 音乐库、收藏歌曲和最近播放也要有批量播放/入队动作，照顾重复听歌和整理队列的高频路径。
- 收藏状态筛选复用现有过滤链路，并进入筛选标签栏；移除标签时要同步 select 状态，避免 UI 和结果分叉。
- 批量播放入口旁应提供歌曲数和总时长，帮助用户判断这次播放会持续多久。
- 连续播放应尽量提前预加载下一首，减少切歌时的等待感。
- 播放状态应尽量同步给 Emby，让服务器端最近播放和进度更接近真实使用。
- 移动端要保留核心路径：搜索、播放、队列、专辑详情。
- 移动端主导航采用底部栏承接首页、音乐库、收藏、队列和设置，横向侧栏只作为平板宽度的过渡形态。
- 移动端底部栏的“更多”承接次级入口，避免把歌单、专辑、艺人、最近播放全部塞进固定底栏。
- “更多”菜单和歌曲操作菜单复用同一个通用 action sheet，但标题、空状态和副标题由调用方传入，避免交互语义混乱。
- 详情页导航要高亮父级入口，例如专辑详情归属专辑，歌单详情归属歌单，避免用户在深层页面失去方向。
- 移动端歌曲行只保留高频收藏和更多入口，低频管理动作放进底部菜单，避免按钮挤压标题和封面。

## UI 角度

- 默认浅色主题，避免纯黑播放器感。
- 内容层级使用：顶部搜索/状态，左侧导航，中部内容，底部播放器。
- 专辑封面是主要视觉资产；没有封面时使用稳定的渐变占位。
- 正在播放页使用大封面和大标题，移动端优先保留播放、收藏、进度和后续队列。
- 播放波形用于强化“正在播放”的即时反馈，但先用 CSS 动画跟随播放状态，不接入跨域音频分析，避免影响 Emby 流播放稳定性。
- 正在播放页用徽标展示格式、播放源和策略；底部播放器只保留一行精简状态，避免日常控制区变重。
- 歌词放在正在播放页右侧，和后续队列同层，既服务沉浸播放，也不挤占歌曲列表。
- 艺人页应进入独立详情，而不是只做隐藏筛选；用户需要看到艺人专辑、艺人歌曲和直接播放入口。
- 歌单页使用独立封面卡片，详情页复用专辑页的信息层级，降低学习成本。
- 收藏按钮使用统一心形控件，卡片上做轻量悬浮，详情页放在播放动作旁边。
- 收藏页用分段区块承接不同实体类型，保持扫读清楚，避免把不同内容混成一个长列表。
- 首页只展示已有数据的区块，避免空内容占用首屏，同时保留前往完整页面的入口。
- 歌曲行保留点击主体播放，但艺人/专辑作为细粒度文本按钮，满足精确导航。
- 当前播放歌曲在所有复用列表中都应有明确播放/暂停指示，用户不该在长列表里只靠背景色猜位置。
- 底部播放器要能一键定位当前歌曲；先在当前页面找，找不到再跳到播放队列，减少边听边浏览时的迷失。
- 歌曲行用小徽标承载 FLAC、MP3 320k 等格式信息，详情页聚合格式摘要，保留音乐客户端的扫读速度。
- 专辑、艺人、歌单、队列和列表页的 meta 文案应显示总时长，用户播放整组内容前需要这个判断信息。
- 专辑详情要尊重 Emby 专辑元数据，歌曲左侧显示碟号/曲号，而不是所有场景都用当前列表序号。
- 播放页的艺人/专辑按钮保持低视觉噪音，默认像文本，悬停和聚焦才强调可点击。
- 返回按钮文案跟随目标页面变化，让用户在点击前知道会回到哪里。
- 控件保持音乐应用语义：播放、随机、队列、排序、刷新、清除筛选。
- 风格、年份和音质筛选放在音乐库头部，和排序同级，筛选状态进入统一筛选条，保证可见可清除。
- 键盘和系统媒体键属于高频播放器能力，不在界面上堆说明文字，但要让熟悉音乐客户端的用户自然可用。
- 卡片只用于专辑、艺人、状态块；页面区块保持干净，不堆叠卡片。

## 开发角度

- 当前仍是无构建静态项目，方便快速迭代和直接部署。
- 服务器预设留在 `src/config.js`，不引入构建时环境变量，保持复制文件即可部署。
- 模块化先拆纯配置、格式化、歌词解析、Emby API helper 和本地存储 helper，避免把 DOM/状态/播放器流程一次性打散。
- PWA service worker 只处理同源静态资源，Emby API、封面和音频流保持网络直连，避免缓存鉴权内容和媒体数据。
- 应用外壳资源优先走网络再回退缓存，避免旧 service worker 长时间拦截登录页脚本。
- 静态资源 URL 带应用版本参数，确保旧 service worker 控制页面时也能请求到最新脚本。
- hash 路由用于安装快捷方式和直达页面，不引入完整路由框架。
- 桌面侧栏、顶部标签和移动底部导航都复用 `data-view` 与 `switchView`，保持单一视图切换源。
- 详情页使用 `getNavigationView` 映射父级导航，高亮逻辑不改变真实 hash 和面板状态。
- 移动端更多入口通过 `isMobileMoreNavigationView` 单独计算 active 状态，避免固定底栏和次级页面状态脱节。
- 设置页聚合浏览器网络、连接测试、PWA、媒体键、窗口标题、队列、最近播放、歌词和诊断信息，降低后续维护和排查成本。
- 统一通知条用于跨视图错误和恢复动作，普通成功状态仍保持轻量短提示。
- API 调用集中在 `embyFetch`、`userItemsPath`、图片 URL、音频 URL 等函数。
- 音乐库分区选择使用 Emby `Views` 中的 music collection，分页请求统一通过 `getLibraryScopeParams()` 注入 `ParentId`，偏好由 `src/storage.js` 按服务器和用户隔离保存，避免初次加载和加载更多使用不同范围。
- 列表密度是全局 UI 偏好，使用 `body.density-compact` 控制所有歌曲列表尺寸；渲染层不分叉，避免每个列表单独维护样式状态。
- 排序字段和方向作为本地偏好保存在 `src/storage.js`；默认方向由当前排序字段推导，手动升序/降序只改变比较结果，不改变筛选范围。
- `src/emby-api.js` 承接请求、同源鉴权头、URL 构建、图片/音频地址构造；`app.js` 只保留薄包装以降低改动面。
- `src/storage.js` 承接 session、设备 ID、设备名、播放模式、音量和最近播放持久化，减少 `app.js` 对 `localStorage` 的直接依赖。
- 播放器用 `playRequestId` 避免旧播放请求误报错误。
- 播放可视化只监听统一播放状态类，不创建 `AudioContext`，后续需要真实频谱时再作为独立增强模块接入。
- 播放源策略保存在本地；自动策略保持直连优先，真实失败才降级到 Emby universal/transcode，手动策略尊重用户选择并提供反向重试入口。
- 播放模式和音量偏好保存在本地，避免每次进入都回到默认控制状态。
- 睡眠定时保存在当前页面运行状态内，刷新后自动失效，避免隔天打开应用仍保留旧倒计时。
- 队列操作保持在前端状态内完成，修改后同步当前曲目索引并刷新活跃行；拖拽排序和按钮排序复用同一条重排路径。
- 队列定位使用当前曲目的 `Id` 在队列列表中查找活跃行，并使用 `CSS.escape` 处理特殊字符 ID。
- 清除已播只裁剪当前曲目前的队列项，把当前曲目设为索引 0；不暂停音频、不重设播放源，也不清除后续待播内容。
- 队列整理复用 `getQueueShuffleStartIndex()` 的范围，和随机重排保持同一行为边界；排序结果保存到本地队列，刷新后保持。
- 歌曲行操作先生成统一 action 定义，桌面直接渲染按钮，移动端复用同一组定义渲染底部操作菜单，避免两套行为分叉。
- 队列状态保存在 `localStorage`，按服务器地址和用户隔离；恢复后只显示当前歌曲，不自动播放。
- 首页恢复卡片复用当前队列状态和保存进度，播放时通过 `playTrack(..., { positionSeconds })` 接回同一条恢复链路。
- 队列本地存储读取 `savedAt` 到状态中；保存和清空队列时同步刷新首页恢复卡片，避免保存时间显示滞后。
- 批量加入队列按 `Id` 去重；空队列时只建立队列和当前歌曲，不自动播放，避免违背用户意图。
- 列表页批量播放复用同一套队列入口；搜索状态下只作用于当前可见结果。
- 歌曲筛选逻辑集中到 `matchesTrackFilters`，音乐库和最近播放共用，减少新增筛选维度时的漏接风险；持久化筛选状态按服务器地址和用户隔离，避免不同 Emby 实例互相污染。
- 风格筛选从已加载歌曲/专辑的 `Genres` 派生，年份筛选从 `ProductionYear` 派生，和搜索、收藏歌曲、批量播放共用过滤结果。
- 音频格式徽标和音质筛选从 Emby `MediaSources` / `MediaStreams` 派生；缺少媒体字段时不展示也不启用筛选，避免制造不可靠信息。
- 下一首预加载使用独立 audio 元素，不影响当前播放元素和错误处理链路。
- 正在播放页复用底部播放器状态，避免出现两个播放状态源。
- 主播放控制必须固定在底部可见区域：左侧承载当前曲目，中间承载播放/进度，右侧承载模式、队列和音量，列表内容滚动不应影响用户找到播放按钮。
- 顶部工具栏保持客户端式轻量排版：品牌/窗口区、主导航、搜索与状态工具分区明确，避免把刷新、退出、状态和搜索挤成同等视觉权重。
- 低频顶栏动作进入独立账号弹窗，头像入口只负责打开账号/连接菜单；图标采用彩色线性 SVG，风格上更接近音乐客户端工具区，同时不依赖外部图标资源。
- 首页避免卡片堆叠式布局，采用简约音乐客户端的信息流：上方是当前连接和快捷播放，状态数据用细线分隔，专辑与歌曲列表直接排布在内容区。
- 移动端保留固定播放能力但压缩播放器高度，底部播放器只保留曲目信息、播放键和进度；队列、设置等低频入口交给移动底栏，避免播放器挤占内容浏览空间。
- 当前曲目的动态强调色来自歌曲元数据确定性映射，统一作用于播放器、进度条、列表高亮和设置诊断；暂不做封面 canvas 取色，避免 Emby 跨域封面污染 canvas。
- 快捷键只在非输入状态生效，避免打断搜索、登录和表单输入。
- Media Session action 做能力探测和异常兜底，浏览器不支持时不影响网页内播放器。
- `document.title` 从播放器元数据和播放状态统一更新，清空会话或没有当前歌曲时恢复应用名，并同步到设置页和诊断信息，避免 PWA 窗口残留旧曲目却难以排查。
- 歌词优先从 Emby 已返回的 `MediaSources` 文本附件/嵌入字段中解析，避免新增不稳定接口依赖；LRC 有时间轴时随播放高亮，TXT 则静态展示。
- 分页加载使用 `StartIndex` 和 `Limit`，并在浏览器端按 `Id` 合并去重。
- 艺人详情优先使用 Emby `ArtistIds`/`AlbumArtistIds` 查询，失败时回退到当前已加载的本地音乐库内容。
- 歌单列表通过 `IncludeItemTypes=Playlist` 渐进加载，歌单详情使用 `ParentId` 普通只读查询，并通过 `StartIndex/Limit` 分页加载歌曲；打开详情时只取第一页，避免超大歌单一次性拖垮 Emby 或前端。
- 创建歌单使用 Emby `POST /Playlists`，传入 `Name` 和 `MediaType=Audio`，成功后合并到本地列表。
- 添加歌曲到歌单使用 Emby `POST /Playlists/{Id}/Items`；从歌单移除使用 `PlaylistItemId` 和 `DELETE /Playlists/{Id}/Items?EntryIds=...`，并保留 `POST /Items/Delete` 兼容兜底。
- 歌单排序使用 Emby `POST /Playlists/{Id}/Items/{ItemId}/Move/{NewIndex}`；优先使用 `PlaylistItemId`，失败时回退歌曲 `Id`，提高不同服务器版本的兼容性。
- 收藏使用 `FavoriteItems` 接口 best-effort 同步，覆盖歌曲、专辑、艺人和歌单；失败时回滚本地状态。
- 收藏页数据直接从本地已加载实体的 `UserData.IsFavorite` 派生，和搜索过滤共用 `applyFilters`。
- 首页预览复用现有渲染器，不单独维护一套卡片逻辑，减少 UI 分叉和状态不一致。
- 歌曲行导航优先使用已加载实体或 Emby 返回的 `AlbumId`/`ArtistItems`，缺少 ID 时降级为名称搜索。
- 详情页记录进入前的 active panel，返回时只切换本地面板，不引入额外路由依赖。
- 最近播放保存在 `localStorage`，只存有限条目，避免无限增长。
- Emby 播放状态上报保持 best-effort，任何失败都不影响本地播放。
- 优先保持状态集中在 `state`，后续如果迁移到 Vue/React，可按同样状态结构拆模块。
- 需要继续拆分：API 层、播放器层、渲染层、状态层；每次拆分必须保持无构建静态部署可验证。
